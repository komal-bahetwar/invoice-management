# Azure Infrastructure as Code

> Bicep templates for provisioning Azure production resources for the Invoice Management API.
> Lives at repo root (`infra/`) to serve both backend and future frontend deployments.

## Overview

The IaC is authored in two tiers:

| Tier | Source | Covers |
|---|---|---|
| **Core** (root files) | Auto-generated by `azd init` from `backend/AppHost.cs` | Azure Container Apps (api + migrator), Azure SQL Database |
| **Supplemental** (`modules/`) | Manually authored Bicep | Service Bus, Key Vault, App Insights, ACR, Redis |

### Why two tiers?

`.NET Aspire`'s `AppHost.cs` already defines the API project and SQL Server. Running `azd init` reads this and auto-generates production-grade Bicep for ACA and Azure SQL — managed identity, autoscaling, private endpoints — all out of the box. Writing these manually would duplicate what `azd` generates.

Supplemental resources (messaging, secrets, observability, registry, caching) are not part of the Aspire topology and are authored as standalone modules under `modules/`. The root `main.bicep` references these modules so a single `azd up` provisions everything.

## Structure

```
infra/
├── README.md
├── main.bicep                    # Orchestrates all resources (azd-generated, references modules/)
├── main.parameters.json          # azd-generated, extended with supplemental params
├── modules/                      # Supplemental Bicep modules (manual)
│   ├── service-bus.bicep
│   ├── key-vault.bicep
│   ├── app-insights.bicep
│   ├── container-registry.bicep
│   └── redis.bicep
└── parameters/                   # Environment-specific overrides
    ├── dev.parameters.json
    ├── staging.parameters.json
    └── prod.parameters.json
```

## Resources

### Core (azd-generated)

| Resource | Azure Service | Details |
|---|---|---|
| **Compute** | Azure Container Apps Environment | Hosts `invoice-api` (API) and `invoice-migrator` (one-shot EF migration job). Scale-to-zero in dev, 2+ replicas in prod. |
| **Database** | Azure SQL Database | Serverless Gen5 (dev/staging), Provisioned Gen5 elastic pool (prod). Schema-per-tenant via Finbuckle. |

### Supplemental (modules/)

| Module | Azure Resource | Purpose |
|---|---|---|
| `service-bus.bicep` | Azure Service Bus | MassTransit messaging (invoice-events, tenant-events topics) |
| `key-vault.bicep` | Azure Key Vault | Secrets: connection strings, signing keys, API keys |
| `app-insights.bicep` | App Insights + Log Analytics | OpenTelemetry traces, metrics, structured logs |
| `container-registry.bicep` | Azure Container Registry | Private Docker image storage, vulnerability scanning |
| `redis.bicep` | Azure Cache for Redis | Dashboard caching, tenant config caching |

## Deployment

```bash
# 1. Generate base infra from backend's AppHost.cs (run from repo root)
azd init --project-path backend

# 2. Deploy (azd reads azure.yaml → provisions infra/ Bicep)
azd up

# Environment switching
azd env new staging
azd env select staging
azd up
```
